# ADR-0009：HITL 修订指令定位增强与恢复值契约扩展

状态：已接受（2026-07-22）。
适用范围：人工中断点主节点（`src/nodes/human_review_gate.py`）、审阅提交服务层（`src/service/task_service.py`、`src/service/app.py`）、章节清单装配段（`src/assembly/context_assembler.py`）。
承接 issue #41（PRD v2）的 T4 分票（issue #49）；restructure 结构重修不在本 ADR 范围（已拆 issue #42 / T5）。
动因：人工修订原本只支持「LLM 解析 + 按章点名」一条路——用户引用原文时无法定位、表达全局不满时要逐章重复、一句话可能意外触发全篇重写、意见含混时系统靠猜。

## 1. 两级定位：确定性优先，LLM 兜底

用户意见引用正文原文时，解析 LLM 只负责把引文逐字摘出（locate=quote 条目），定位由程序两级完成：

- 第一级（确定性）：对引文与各章草稿正文做归一化（剔除素材角标与全部空白）后的子串匹配；唯一命中直达目标章，命中结果覆盖 LLM 的章节判断。
- 第二级（LLM 兜底）：未命中或多章命中时，回退解析调用中 LLM 按章节摘要给出的 `target_chapter_id` 判断——为此章节清单装配段（chapter_list）扩为每行「id 标题：草稿摘要」。

两级都失败即定位失败，转含混回问（第 4 节），不猜测。
不为定位单开第二次 LLM 调用：解析与定位判断合并在同一次调用产出结构化条目，路由归结全部由程序确定性完成（延续「LLM 只产结构化数据、程序选边」原则）。

## 2. 全局意见程序侧扇出

意见针对全篇普遍适用时，LLM 标记 locate=global，程序侧确定性扇出为大纲每章一条同型修订指令，逐章执行。
扇出产物与显式章节条目按（章节, 类型, 指令）去重保序。
全局扇出必然触及全部章节，因此天然落入大扇出确认（第 3 节）——这是有意行为：全篇重写必须经人工回显确认。

## 3. 大扇出确认与恢复值契约扩展

一轮解析归结出的指令触及的章节数严格超过大纲一半（2 × 受影响章数 > 大纲章数）时，不直接执行：携解析清单（`pending_confirmation`：受影响章节、大纲总章数、逐条指令回显）重新中断待人工确认，复用安全汇点重新中断模式。

恢复值契约由两动作扩为三动作：`finalize` / `revise` / `confirm`。
`confirm` 仅在存在待确认清单时合法，确认后按清单执行；用户也可改提 `revise`（作废清单按新意见重新解析）或 `finalize`。
动作全集以 `human_review_gate.RESUME_ACTIONS` 为单一事实源，服务层恢复值唯一产地 `task_service.build_resume_value` 与 HTTP 层 `ReviewRequest` 与之同形，契约测试（`tests/service/test_review_resume_contract.py`）锚定三处一致。

节点重放语义的取舍：LangGraph 恢复时节点从头重放、既有 interrupt 按序返回历史恢复值。
若解析 LLM 调用在重放中重新执行，真实 LLM 的非确定性会使重放解析出的清单偏离用户确认的那份——可能执行一份用户从未回显确认的清单，甚至因重放解析恰好不再触发大扇出而绕过确认直接执行。
为此意见解析的 LLM 调用包在 LangGraph `task`（durable execution）里、闭包捕获 `llm_factory` 保留测试桩注入点：调用结果随任务写入 checkpoint，重放时直接取缓存、不重复调用，保证 confirm 执行的清单严格等于确认中断时回显给用户的那份。
任务只返回原始 JSON 负载（纯标量/字典/列表，可序列化），定位归结 `resolve_directives` 是确定性纯函数、在任务外重放执行，其结果随负载缓存天然一致。
重放后程序仍重新校验清单与确认条件（重放负载若被人工改成回问或解析失败形态，confirm 按契约不符重新中断），绝不执行未经确认的清单。
回归测试 `test_confirm执行的是确认时回显的清单_重放不因LLM非确定而漂移` 以两份不同应答锚定此性质：第二份永不被消费、执行的是第一份。

附带修正：LangGraph 1.x 下同节点内重新中断后 `snapshot.next` 为空而中断仍挂在任务上，服务层 `_at_interrupt` 判定去掉了「next 非空」前置条件——否则错误重提、回问、确认三种重新中断在 HTTP 层全部误判 409。

## 4. 含混回问：不猜测

意见含混（LLM 标记 locate=unclear 并给出回问问题）或引文两级定位失败（程序生成回问）时，携 `clarification_questions` 重新中断回问用户。
只要存在任何回问问题，本轮不执行任何已解析出的指令——部分执行会让用户对「哪些意见生效了」失去判断。
章节 id 幻觉（locate=chapter 但 id 不在大纲）仍按模型噪声静默丢弃，不转回问：它不是用户表达含混，安全汇点的「解析不出任何有效指令」错误重提兜底。

## 后果

- 恢复值契约是对外 REST 契约的一部分，`confirm` 动作与 `review_required` 载荷新字段（`pending_confirmation`、`clarification_questions`）随 `docs/api.md` 发布，调用方需按载荷字段区分三种重新中断。
- 两章大纲的演示与端到端链路中，触及双章的混合意见现在多一步确认往返；demo 脚本与相关 E2E 已同步。
- 大扇出阈值（严格超过一半）按 PRD 拍定为固定规则，不设环境变量；单章大纲的任何修订都会触发确认，属规则的字面推论，暂不特判。
