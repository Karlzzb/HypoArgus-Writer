# ADR-0001：子智能体真实现落地的四条硬约束

状态：已接受（2026-07-18）。
适用范围：`agents/` 内 search_agent 与 rewriter_loop 的真实现开发，以及未来新增的任何子智能体。
与 PRD 的关系：`PRD.md` 为已完成的历史版本，与本 ADR 表述不一致处（子智能体内部事件的暴露方式、任务包字段清单）以本 ADR 为准。

## 背景

当前两个子智能体均为打桩：瞬时完成、零副作用、完全确定性。
排查确认桩掩盖了两类问题：节点整体重跑导致子智能体调用（外部检索、LLM 花费）无条件重复执行；子智能体运行中对外完全静默，前端看不到"正在写第几章"。
根因相同：章级循环长在编排节点函数体内，对 checkpoint 与事件流都不可见。

## 约束

### 1. 章级产物必须增量落 checkpoint

编排节点不得再把多章循环的中间产物（素材、草稿）攒在局部变量、节点结束才整体写 state。
每完成一章，产物必须进入主图 state 并被 checkpoint 覆盖，使中断后 resume 不重复已完成章节的外部调用与 LLM 花费。
实现手段不限（图内循环边、`Send` 分发、或节点内显式提交），但"崩溃重跑只损失当前进行中的一章"是验收标准。

### 2. 事件契约必须带业务上下文与运行中进度

`subagent_start`/`subagent_end` 载荷不得只有 unit 名：至少携带 chapter_id 与调用模式（首写/修订/终审回退）。
子智能体真实现的内部关键步骤（每轮检索、每段生成、每次自检）必须能向外发进度事件——通过任务包或 `run` 签名传入 EventHook，复用既有 dispatcher → emitter → SSE 旁路，不新建事件通道。
验收标准：writing_orchestrator 执行期间，SSE 前端能实时看到"正在写第几章、进行到哪一步"。

### 3. 保持非子图边界

子智能体仍是任务包 dict 进/出的黑盒异步调用，禁止改造为 LangGraph 子图；编排器只依赖 `agents/contracts.py` 的 `Subagent` 协议。
观测与断点问题一律通过约束 1、2 的机制解决，不得以"共享 checkpoint"为由突破此边界。

### 4. 中断场景必须有测试覆盖

真实现合入前，必须新增 E2E 用例：进程 kill 于子智能体执行中途（某章完成、下一章未完成）后 resume。
用例须断言：已完成章节不重复执行（幂等/成本）、事件成对且父子链正确、最终产物与不中断路径等价。
仅在桩上通过的验收不算通过——涉及时序与副作用的行为必须在真实现（或有耗时/副作用模拟的桩）上验证。

## 后果

- 契约（`agents/contracts.py`）与两个编排节点的逐章循环需要为约束 1、2 开口子，属预期改动。
- 章级落 state 会增加 checkpoint 写入频次，换取崩溃重跑成本从"整节点"降为"单章"，符合项目优先健壮性的取舍。
- 违反任一约束的子智能体实现不予合入。
