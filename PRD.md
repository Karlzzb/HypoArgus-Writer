# HypoArgus-Writer 产品需求文档（终版）

本文档是项目唯一有效 PRD。
领域术语以根目录 `CONTEXT.md` 词汇表为准。

## 问题陈述

专业写作场景普遍存在以下问题：

1. 文章缺乏结构化论证体系，观点没有可验证的依据支撑。
2. 引用不严谨，来源不可溯源，或在文末集中堆砌而非放在观点原位。
3. 长文章章节之间逻辑割裂，前后不连贯。
4. 多轮修改迭代后，智能体上下文被污染，出现「失忆」，用户感觉智能体记不得之前的要求。
5. 智能体执行流程是黑盒，外部系统无法还原其运行拓扑，用户无法观察进度与分支走向。
6. 不同环节对模型能力要求不同，但现有方案只能全局共用一套大模型配置。

## 解决方案

搭建一套纯 LangGraph 单一技术栈的工业级结构化写作后端服务，只提供后端能力，不含前端页面。

核心机制：

- 由 5 个 LangGraph 主节点组成刚性流水线：framework_orchestrator（论证框架生成）、reference_orchestrator（检索调度）、writing_orchestrator（串行写作总控）、citation_validator（引文终审门禁）、human_review_gate（人工中断点与迭代路由）。
- 论证体系采用三层结构：章节 1—n 论点，论点 1—N 假说；假说必须可证伪、可检索验证，是检索任务的直接驱动源。
- 假说生成继承六角度发散框架（假设、失效模式、边界条件、竞争解释、预言、反事实），随后按证据可检索性筛选，各层数量上限可配置。
- 两个业务子智能体 search_agent（检索与素材相关性校验）、rewriter_loop（章节写作与循环润色）以黑盒适配层接入，本期打桩，后续迁移真实现。
- 引用采用「正文角标 + 结构化引文库」分离方案，引文对账以纯程序逻辑为主、LLM 语义核查为辅。
- 用户审阅意见由 LLM 解析为结构化修订指令列表，纯改写与补充佐证两类分支可在同一轮迭代内混合执行，无限循环直至定稿。
- context_assembler 组件在每次 LLM 调用前从 State 结构化装配输入包，保证多轮迭代不失忆、不污染用户原始上下文。
- 全部 7 个运行单元（5 主节点 + 2 子智能体）支持通过 `.env` 独立配置 LLM 参数，未配置的回落全局缺省。
- 对外提供两条隔离的 SSE 通道：业务数据流与 `graph_event` 可视化事件流，事件信封携带父子链路 ID，供前端自主还原动态 DAG 拓扑。
- 全流程状态经 Postgres 存档器持久化，支持断点续跑与历史版本回滚；可观测接入自建 Langfuse。

## 用户故事

1. 作为专业撰稿人，我想输入写作需求后系统自动识别文章品类并匹配模板，这样我无需手动搭建文章框架。
2. 作为专业撰稿人，当我的需求匹配不上任何模板时，我想系统自动转入自由结构模式生成大纲，这样流程不会中断报错。
3. 作为专业撰稿人，我想大纲严格沿用模板的一二级标题骨架并按我的需求实例化，这样产出符合行业规范格式。
4. 作为专业撰稿人，当模板填充变量我没有提供时，我想系统用醒目占位标记继续写作而不是追问阻塞，这样首稿能一次性产出，我在审阅时再补充。
5. 作为专业撰稿人，我想每个章节拥有明确的论点和可证伪的假说，这样写作完全有据可依。
6. 作为专业撰稿人，我想假说从六个不同角度发散生成并去重筛选，这样论证体系不会塌缩成同一观点的重复表述。
7. 作为专业撰稿人，我想所有写入正文的素材都经过检索智能体的相关性校验，这样正文不会出现不可靠信息。
8. 作为专业撰稿人，我想章节严格串行写作且自动承接上一章摘要，这样长文前后逻辑连贯不割裂。
9. 作为专业撰稿人，我想引用以角标形式标注在对应观点的段落原位，这样每个观点都能就地溯源。
10. 作为专业撰稿人，我想全文引文经过独立终审对账，这样不会出现无来源的标注或张冠李戴的引用。
11. 作为专业撰稿人，我想提交一条混合了多种诉求的修改意见后，系统自动拆解为逐章节的修订指令并区分纯改写与补充佐证，这样修改精准落到位。
12. 作为专业撰稿人，我想修改只作用于指定章节，这样不会出现全文无差别重写。
13. 作为专业撰稿人，我想迭代多轮后系统仍记得我历次提过的要求和偏好，这样我不必重复交代。
14. 作为专业撰稿人，我想修订后的章节自动重新通过引文核查，这样迭代不会绕过质量门禁。
15. 作为专业撰稿人，当引文核查多次失败时，我想系统带着警告把决定权交给我而不是无限重试，这样流程永远不会卡死。
16. 作为专业撰稿人，我想任务可以随时暂停重启，服务崩溃不丢进度，这样长文写作有安全感。
17. 作为专业撰稿人，我想回退到任意历史写作版本，这样可以撤销不满意的迭代。
18. 作为系统架构负责人，我想为每个主节点和子智能体在 `.env` 中独立配置模型名称、接口地址、密钥，这样不同环节可选用不同能力规格的模型。
19. 作为系统架构负责人，我想未单独配置的运行单元自动回落全局缺省模型配置，这样不必为每个单元填写全部参数。
20. 作为系统架构负责人，我想论点、假说的数量上限和重试次数都是配置项，这样可以按业务场景调节成本与深度。
21. 作为前端可视化开发人员，我想通过独立 SSE 通道持续收到标准化运行事件流，这样可以自主搭建动态 DAG 可视化大屏。
22. 作为前端可视化开发人员，我想每条事件携带全局唯一 ID、追踪 ID、父级事件 ID 和时间戳，这样可以自动拼接完整执行拓扑。
23. 作为前端可视化开发人员，我想事件覆盖节点启停、分支流转、循环迭代、子智能体调用、状态快照、模型配置、进度七类，这样黑盒执行过程完全透明。
24. 作为前端可视化开发人员，我想通过查询参数按事件类型订阅过滤，这样可以只接收关心的事件、控制数据量。
25. 作为外部调用系统，我想在创建任务时传入自己的会话标识并在事件流中按它过滤，这样能在不依赖本系统鉴权的前提下隔离不同用户的会话。
26. 作为技术运维人员，我想全链路 LLM 调用自动上报自建 Langfuse，这样可以审计每次调用的输入输出与耗时成本。
27. 作为后续迁移子智能体的开发人员，我想接口规范与打桩实现已就位且主流程可端到端空转，这样迁移时只需让真实现满足接口规范即可替换。

## 实施决策

### 技术栈与基建

- 纯 LangGraph（1.x）单一编排技术栈，以纯库形态嵌入自建 FastAPI 服务，不使用 LangGraph Agent Server。
- 状态持久化使用 LangGraph 官方 Postgres 存档器（Checkpointer），连接串由环境变量提供；断点续跑、人工中断点恢复、版本回滚全部构建在存档器之上。
- 可观测使用自建 Langfuse，通过官方回调接入；不使用 LangSmith。
- ulmen-langgraph 压缩仅作为实验性可选的序列化反序列化器（serde）接入存档器，默认关闭；关闭开关后历史存档必须仍可读取，做不到则不启用。
- 运行环境为既有 conda 环境（Python 3.11），依赖以 `pyproject.toml` 为唯一事实源，uv 作为安装器；源码平铺于 `src/` 目录。

### 模型配置

- 7 个运行单元各自支持在 `.env` 中以「单元名前缀 + `LLM_MODEL` / `LLM_BASE_URL` / `LLM_API_KEY`」独立配置；未配置项回落到无前缀的全局缺省变量。
- 所有模型统一按 OpenAI 兼容接口封装，`base_url` 止于兼容根路径。
- 不做模型降级、多套配置方案切换、运行时热更新。

### 论证体系与假说生成

- 层级关系：章节 1—n 论点，论点 1—N 假说。
- 假说生成提示词继承既有六角度框架与可证伪性检验、差异去重机制；输入从单一主张改为「章节语境下的单个论点」。
- 生成后按「证据可检索性」筛选，被筛掉的假说不进入 State。
- 数量上限均为配置项：每章论点数、每论点假说数、全文假说总数（缺省 4 / 3 / 60）。

### 模板与大纲

- 模板库为本地 `docs_templates/` 目录，品类识别由 LLM 依据模板索引完成。
- 大纲生成方式：解析模板一二级标题层级为骨架，LLM 仅做标题实例化与不适用章节裁剪，不得自由重造结构。
- 匹配置信度低时转入自由结构模式：跳过模板骨架直接生成大纲，其余流程完全一致。
- 模板填充变量缺失时写入醒目占位标记继续写作，不阻塞流程，由用户在审阅轮通过修订指令补充。

### 引用体系

- 正文只嵌轻量角标（素材 ID），完整引文信息（来源、链接、摘录、关联假说、校验结论）全部存于 State 结构化引文库。
- 引文对账为纯程序逻辑：提取全文角标与引文库互查（无来源的标注、未被引用的素材、跨章误引）；citation_validator 的 LLM 只负责语义核查（标注位置与观点是否对应）。
- 双层校验：rewriter_loop 单章自检 + citation_validator 全局终审，缺一不可。
- 最终交付时可从引文库按任意书目格式渲染，格式与内容解耦。

### 流程闭环

- 主流程：需求提交 → 框架生成 → 逐章检索 → 串行写作 → 引文终审 → 人工中断点 → 修订迭代循环。
- citation_validator 终审失败时定向回退 writing_orchestrator 只重写不合格章节；重试次数可配置（缺省 2），超限不死循环，携带未决引文警告强制进入 human_review_gate 交人工裁决。
- human_review_gate 的 LLM 职责是意见解析器：将用户一次提交的自然语言意见拆解为修订指令列表，每条形如 {目标章节, 类型: 纯改写|补充佐证}；两类分支同轮混合执行。
- 修订指令执行完毕后统一汇入 citation_validator 做增量核查（只重审本轮被修改章节），再回到人工中断点开启下一轮。
- human_review_gate 是全流程唯一安全汇点：任何机器环节失败若干次后都塌缩到这里，保证系统永不转死。
- 全局状态机枚举：IDLE、FRAMEWORK_BUILDING、REFERENCE_FETCHING、ARTICLE_WRITING、CITATION_CHECKING、AWAIT_USER_REVIEW、FINISHED、ERROR_FAILED。

### 上下文装配（context_assembler）

- 每次 LLM 调用的输入由 context_assembler 从 State 现场装配，禁止透传原始累积历史。
- 统一入口 `assemble(state, unit)`，差异收敛于按单元注册的装配配方：每份配方 = 一组提取器（State → 内容段的纯函数，可跨配方复用）+ 专属 token 预算与保留策略。
- 压缩只在超阈值时发生：前文摘要链过长做「摘要的摘要」；修订台账保最近 K 轮原文 + 更早轮次一句话摘要；阈值可配置。
- 用户每轮意见解析后追加进 State 修订台账全量持久化，装配时按需注入，保证多轮迭代不失忆。
- 对子智能体零侵入：装配发生在调用方边界，子智能体只收到装配好的任务包。

### 子智能体接入（本期打桩）

- 调用形态为黑盒异步可调用的适配层，不强制改造为 LangGraph 子图；子智能体启动/结束事件由适配层发出。
- search_agent 按章节批量调用：入参一次给整章假说列表，出参素材逐条回链假说 ID。
- rewriter_loop 统包首写与纯改写两种模式，入参带模式标记。
- 接口规范（来自设计定稿，字段即决策）：

```
search_agent.run(任务包) -> 检索结果
  任务包:   chapter_id, hypotheses[{id, text, refute_condition}], genre, existing_materials_digest
  检索结果: materials[{id, hypothesis_id, source, excerpt, relevance_score, verdict: pass|fail}]

rewriter_loop.run(任务包) -> 改写结果
  任务包:   mode: draft|revise, chapter_spec{id, title, points, hypotheses},
            materials[], prev_chapter_summary, revision_directives[]?, current_text?
  改写结果: chapter_text(含原位角标), chapter_summary, self_check{citations_ok, issues[]}
```

- 本期实现为打桩：返回结构合规的模拟数据，保证主流程端到端空转；真实现迁移时按接口规范替换。

### 对外服务与事件流

- 自建 FastAPI 提供：任务创建/恢复/回滚 REST 接口、业务 SSE 通道、`graph_event` 可视化 SSE 通道，两条通道隔离。
- 事件源为 LangGraph 原生细粒度事件流接口，统一封装为事件信封：

```
{ event_id, trace_id, session_id, thread_id, parent_id, ts, type, unit, payload }
type ∈ { node_start, node_end, node_error, gate_blocked, gate_resumed,
         branch_taken, loop_iteration, subagent_start, subagent_end,
         state_snapshot, llm_config_used, progress }
```

- `thread_id` 是写作任务标识（一篇文章一个）；`session_id` 是外部调用方传入的会话标识，本系统只透传不鉴权，事件流支持按它过滤实现调用方侧会话隔离。
- 状态快照事件只含元数据（状态机枚举值、章节完成度、迭代轮次、素材计数），绝不含正文全文。
- 可视化通道支持按事件类型的查询参数订阅过滤，本期即实现。

### State 结构

- WritingAgentState 包含：用户原始意图与身份、文章品类与模板标识、大纲、逐章论点与假说、结构化引文库、逐章正文与摘要链、修订台账、状态机枚举值、迭代轮次、`execution_trace_id`、`current_node_llm_config`。

## 测试决策

- 只测外部行为，不测实现细节。
- 最高测试接缝为 FastAPI HTTP 层：创建任务 → 消费两条 SSE 流 → 提交审阅意见 → 验证闭环流转与最终产物，构成端到端主干测试。
- 第二道接缝是统一 LLM 调用封装层：测试时注入确定性的假 LLM，使 5 个主节点的行为可复现；子智能体天然以打桩形态存在，即是现成的测试替身。
- 纯逻辑模块直接单元测试：引文角标对账、装配配方与提取器、配置读取与回落、模板骨架解析。
- 本项目为全新代码库，无既有测试先例；测试基础设施随首个模块建立。

## 范围外

- 任何前端页面与可视化渲染实现。
- 鉴权、权限管理、密钥托管设施。
- 模型降级容错、多套模型配置方案切换、配置热更新。
- search_agent、rewriter_loop 的内部真实现（在别处，后续单独迁移）。

- 高并发压测与模板热更新。
